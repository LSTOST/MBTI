## 2. 产品定义

### 2.1 产品一句话

一个基于 `MBTI + 太阳星座` 的灵魂伴侣报告产品，用户输入基础资料并完成 MBTI 测试后，获得人格画像、理想对象画像、匹配对象类型推荐和付费 AI 深度关系分析。

### 2.2 产品本质

这不是婚恋平台，也不是严肃心理测评工具。

第一版本质上是一个：

- 强身份认同的测试产品
- 高传播性的结果报告产品
- 轻付费解锁的内容型情感产品

### 2.3 产品定位

面向对人格测试、星座、情感关系内容感兴趣的年轻用户，提供低门槛、强反馈、可分享、可付费的灵魂伴侣报告。

### 2.4 V1 不做什么

- 不做真人匹配
- 不做聊天或社交关系链
- 不做完整占星星盘
- 不做 AI 主导匹配打分
- 不做多端并行发布

## 3. 背景与机会

### 3.1 用户真实动机

用户并不是来获得科学结论，而是来获得三个东西：

1. 一个能解释自己的身份标签
2. 一个能讨论“我适合什么样的人”的关系答案
3. 一个可截图、可转发、可引发社交互动的结果

### 3.2 市场判断

成熟产品证明了三件事：

1. 人格测试结果比测试过程更重要
2. 关系解读比“配不配”更有内容价值
3. 传播和付费都建立在“结果够像你、够像在说你恋爱里的毛病”之上

### 3.3 项目目标

先验证“测试 -> 报告 -> 分享 -> 付费 -> AI 深度解读”这条闭环，再决定是否延伸到双人合盘或社区。

## 9. 功能需求

### 9.1 首页

#### 目标

#### 必备元素

### 9.2 基础信息录入

#### 字段

- `gender`
- `birthDate`

#### 规则

- 昵称必填，允许 1-12 字
- 性别为文案定制字段，不参与底层匹配分
- 出生日期用于计算太阳星座

### 9.3 MBTI 测试

#### 题量

建议 `32-40 题`，默认 `36 题`

#### 题型

- 一屏一道题
- 两端语义陈述
- 中间 5 档选择

#### 维度分布

36 题在四个维度上的分配：

| 维度 | 题数 | 说明 |
|------|------|------|
| `E/I`（外向 / 内向） | `9 题` | 社交能量来源、独处偏好 |
| `S/N`（感觉 / 直觉） | `9 题` | 信息接收方式、关注细节 vs 大局 |
| `T/F`（思考 / 情感） | `9 题` | 决策依据、逻辑 vs 价值观 |
| `J/P`（判断 / 感知） | `9 题` | 生活组织方式、计划 vs 灵活 |

#### 出题原则

- 题目必须使用恋爱 / 关系场景化表述，不使用通用职场 / 学业场景
- 两端语义陈述必须等长度、等吸引力，避免一端明显"更好"
- 禁止引导性用语（如"你是不是经常……"暗示某端更正常）
- 每道题只测一个维度，不做交叉测量
- 难度均匀，不在某个维度集中放难题

#### 得分算法

每道题 5 档选择，赋分规则：

| 选项位置 | 左端得分 | 右端得分 |
|---------|---------|---------|
| 最左（强烈偏左） | `+2` | `0` |
| 偏左 | `+1` | `0` |
| 中立 | `0` | `0` |
| 偏右 | `0` | `+1` |
| 最右（强烈偏右） | `0` | `+2` |

每个维度 9 题，满分 `18 分`。取得分高的一端作为该维度结果字母。

维度倾向强度计算：

`倾向强度 = |左端总分 - 右端总分| / 18 * 100%`

倾向强度用于 AI 深度解读中的表述语气调整（强倾向用确定性语言，弱倾向用"你可能偶尔会……"）。

#### 输出

- `E/I`
- `S/N`
- `T/F`
- `J/P`
- 16 型人格
- 四维倾向强度百分比

### 9.7 付费 AI 深度解读

#### 核心定义

AI 不是决策层，而是增强层。它只在灵魂伴侣报告已经生成且用户已付费后，基于结构化摘要生成一次性的深度表达内容。

#### AI 负责什么

- 把结构化结论写成更像真人关系分析的文本
- 加强代入感、冲突解释、关系推进建议
- 提供更有“私人顾问感”的语言表达

#### AI 不负责什么

- 不负责底层匹配计算
- 不改变 MBTI、星座、最佳匹配结果
- 不从原始答题自由生成结论

#### 触发方式

用户已付费查看完整灵魂伴侣报告后，点击按钮：

`生成我的 AI 灵魂伴侣深度解读`

#### 生成内容结构

1. 你的灵魂伴侣核心驱动力
2. 你在关系中的隐藏需求
3. 你为什么容易被某类人吸引
4. 你在暧昧期最容易踩的坑
5. 适合你的关系推进节奏
6. 私人顾问式总结

#### 生成策略

- 首次点击时生成
- 生成结果落库缓存
- 后续重复查看直接读取缓存
- 用户手动刷新页面不重新生成

#### 失败兜底

- AI 生成失败时，完整灵魂伴侣报告仍可查看
- 用户可点击重试
- 重试次数受限并记录失败原因

### 9.8 支付

#### 支付策略

第一版采用单次购买完整灵魂伴侣报告。

#### 付费权益

- 解锁完整灵魂伴侣报告
- 解锁 AI 深度解读能力
- 解锁高质量分享海报

#### 支付接入方案

| 渠道 | 方式 | 优先级 | 说明 |
|------|------|--------|------|
| 微信支付 | JSAPI 支付（微信内）/ H5 支付（浏览器） | P0 | 核心传播渠道在微信生态，必须首批支持 |
| 支付宝 | H5 支付 | P1 | 覆盖非微信流量 |

#### 支付技术流程

1. 前端点击付费按钮，调用 Server Action 创建订单
2. 服务端生成预支付订单，返回支付参数
3. 前端唤起支付 SDK
4. 用户完成支付后，支付平台回调服务端
5. 服务端验证签名，更新订单状态为 `paid`
6. 前端轮询订单状态（间隔 `1s`，最多 `30s`），确认支付成功后刷新页面

#### 订单状态机

`created` → `paying` → `paid` → `fulfilled`

异常分支：

- `created` → `expired`（超过 `15 分钟` 未支付）
- `paying` → `failed`（支付平台返回失败）
- `paid` 但回调延迟：前端轮询兜底 + 服务端定时任务补偿查单

#### 退款策略

V1 不做自助退款。如遇争议，通过客服手动处理。需记录完整订单日志便于排查。

#### 支付后体验要求

- 支付成功即时反馈
- 不等待 AI 返回
- 先展示完整灵魂伴侣版，再引导用户点击 AI 解读

### 9.10 用户身份与会话方案

#### V1 身份策略

V1 不做注册登录，采用匿名用户 + 设备绑定方案：

1. 用户首次访问时，服务端生成唯一 `userId`，写入 `HttpOnly Cookie`（有效期 `90 天`）
2. 同时在客户端 `localStorage` 存储 `userId` 作为备份标识
3. 所有答题记录、报告、订单均绑定此 `userId`

#### 设备丢失与找回

- V1 不支持跨设备找回，在付费前以弱提示告知用户"报告绑定当前设备"
- V1.5 可扩展手机号绑定能力，支持换设备后找回已购报告

#### 会话安全

- `userId` 不暴露在 URL 中
- 报告页 URL 使用独立的 `reportSlug`（短随机串），不等于 `userId`
- 分享链接指向公开摘要页，不暴露完整灵魂伴侣报告数据

### 9.11 测试中途退出与恢复

#### 存储策略

- 答题进度实时保存至 `localStorage`（每选一题即写入）
- 同时每 5 题向服务端同步一次进度快照

#### 恢复规则

| 场景 | 行为 |
|------|------|
| 关闭页面后 `24 小时内` 重新访问 | 弹出恢复提示："你有一份未完成的测试，是否继续？" |
| 超过 `24 小时` | 进度过期，需重新开始 |
| 清除浏览器数据 | 本地进度丢失，但可从服务端恢复（如 Cookie 未失效） |
| 用户选择"重新开始" | 清除旧进度，创建新的 `attemptId` |

#### 恢复体验

- 恢复后直接跳转到上次未答题目
- 顶部进度条显示已完成比例
- 不展示之前答过的题目详情

## 10. 灵魂伴侣匹配引擎

### 10.1 匹配计算总原则

- 可解释
- 可调参
- 可复用
- 可控输出

### 10.2 计算方式

总匹配分建议：

`总匹配分 = MBTI 匹配分 * 0.65 + 星座匹配分 * 0.35`

### 10.3 MBTI 匹配

- 建立 `16 x 16` 兼容矩阵
- 输出匹配分、关系标签、解释骨架
- 给出 `最佳匹配`、`高热吸引`、`高风险冲突` 三类关系

### 10.4 星座匹配

- 基于太阳星座建立 `12 x 12` 兼容矩阵
- 用于增强情绪氛围、节奏感和关系风格描述

### 10.5 结果组装

匹配引擎输出结构化对象，不直接拼最终文案。

示例输出字段：

- `mbtiType`
- `sunSign`
- `loveStyleLabel`
- `bestMatch`
- `topMatches[]`
- `highRiskMatches[]`
- `strengths[]`
- `conflicts[]`
- `advice[]`

## 11. AI 深度分析能力定义

### 11.1 目标

把灵魂伴侣报告从“像说明书”升级成“像有人在认真分析你的关系模式”。

### 11.2 内容风格

- 犀利
- 具体
- 有关系感
- 不说鸡汤
- 不堆抽象形容词

### 11.3 内容边界

- 只围绕已生成的灵魂伴侣报告展开
- 不能引入与测评结论冲突的描述
- 不能声称科学准确或保证找到真爱

### 11.4 输入要求

AI 只读取结构化摘要：

- 用户画像摘要
- MBTI 结论
- 太阳星座
- 匹配结果摘要
- 风险点摘要
- 建议摘要

不直接读取原始答题细项，避免生成无约束脑补内容。

### 11.5 缓存要求

- 每个已支付报告仅生成 1 份主 AI 报告
- 二次访问读缓存
- 若内容不存在且状态为失败，可触发重试

### 11.6 成本控制

- 限制 prompt 长度
- 限制输出长度
- 按报告类型配置 token 上限
- 记录每次调用 token、耗时、成功率、失败原因

### 11.7 Prompt 模板规范

#### System Prompt 骨架

```txt
你是一位专业的亲密关系分析师。你的任务是基于用户的人格测试结果，生成一份灵魂伴侣向深度关系解读报告。

## 你的风格
- 犀利、具体、有洞察力
- 像一个认识用户很久的朋友在认真分析 TA 的感情模式
- 不说鸡汤，不堆空泛形容词，不假装科学权威
- 用"你"直接称呼用户，制造私人对话感

## 你的边界
- 只基于提供的结构化数据进行解读，不自由发挥
- 不承诺"科学准确"或"保证找到真爱"
- 不输出模棱两可的"你有时候这样有时候那样"
- 所有结论必须与灵魂伴侣报告一致，不得矛盾

## 输出格式
按以下 6 个模块输出，每个模块 80-150 字：
1. 灵魂伴侣核心驱动力
2. 关系中的隐藏需求
3. 为什么容易被某类人吸引
4. 暧昧期最容易踩的坑
5. 适合的关系推进节奏
6. 私人顾问式总结

## 语气调节
根据维度倾向强度调整确定性语气：
- 强倾向（>70%）：使用确定性表述，如"你是一个……"
- 中等倾向（40-70%）：使用倾向性表述，如"你大多数时候倾向于……"
- 弱倾向（<40%）：使用可能性表述，如"你可能在某些时候会……"
```

#### User Prompt 骨架


## 12. 技术架构

### 12.1 总体架构

采用单体应用架构：

- 前端、服务端接口、业务逻辑统一在一个代码仓中
- 核心能力模块化拆分，不做微服务

### 12.2 技术选型

- 前端框架：`Next.js 15`
- UI 框架：`React 19`
- 语言：`TypeScript`
- 样式：`Tailwind CSS v4`
- 动效：`Motion`
- 服务端：`Route Handlers / Server Actions`
- 数据库：`PostgreSQL`
- ORM：`Prisma`
- 缓存 / 限流：`Redis`
- 埋点分析：`PostHog`
- AI 接入：`OpenAI Responses API`
- 文件存储：`S3 / R2`
- 部署：`腾讯云 / 阿里云`

### 12.3 逻辑模块

#### Quiz Engine

- 管理题库
- 记录答题进度
- 计算 MBTI 四维得分

#### Zodiac Engine

- 根据出生日期计算太阳星座

#### Compatibility Engine

- 根据 MBTI 和星座输出匹配结果

#### Report Composer

- 组装免费报告与完整灵魂伴侣报告

#### Paywall Service

- 管理付费状态
- 控制报告可见范围

#### AI Analysis Service

- 基于结构化结果生成 AI 深度解读
- 管理生成状态、缓存和重试

#### Share Renderer

- 生成分享海报和 OG 图
- 技术方案：使用 `@vercel/og`（Satori）在 Edge Runtime 将 React 组件渲染为图片
- 海报模板以 React 组件编写，支持动态数据注入
- 生成后的图片上传至 S3 / R2，返回 CDN URL
- 同一报告的海报缓存复用，不重复生成

### 12.4 数据流

1. 用户提交基础资料
2. 用户完成 MBTI 测试
3. `Quiz Engine` 计算 MBTI
4. `Zodiac Engine` 计算太阳星座
5. `Compatibility Engine` 生成结构化匹配结果
6. `Report Composer` 输出免费报告和完整灵魂伴侣报告
7. 用户支付后解锁完整灵魂伴侣报告
8. 用户点击 AI 解读后，`AI Analysis Service` 读取结构化摘要并生成 AI 报告
9. AI 报告保存后供后续直接读取

## 14. 视觉设计风格

### 14.1 设计总原则

高级感的底层公式：

`70% 排版留白 + 20% 顶级素材（插画 / 3D / 动效） + 10% 克制色彩`

所有视觉决策围绕六条核心铁律展开：

1. 先做信息层级，不先做组件
2. 先做大构图，不先堆卡片
3. 减字、减控件、减装饰
4. 统一视觉语法
5. 用对比制造高级感
6. 动效只服务层级，不做装饰

### 14.2 信息层级规范

每一屏必须在 3 秒内让用户看懂"这一屏干什么"。按优先级定义每一屏的视觉权重分配：

#### 首页

| 层级 | 内容 | 视觉权重 |
|------|------|---------|
| 第一眼 | 核心视觉锚点（主插画 / 3D 元素） | 占据首屏 50% 以上面积 |
| 第二眼 | 一句话价值主张 | 大标题，字号 28-32pt |
| 最重要动作 | 开始测试按钮 | 唯一高饱和色块，底部固定或视觉重心区 |
| 弱化 | 产品名、信任说明 | 小字号，低对比度，不抢注意力 |

#### 测试页

| 层级 | 内容 | 视觉权重 |
|------|------|---------|
| 第一眼 | 当前题目文案 | 居中大字，页面绝对主角 |
| 第二眼 | 选项区域 | 清晰可点击，但视觉弱于题目 |
| 最重要动作 | 选中某个选项 | 选中态有明确反馈（色彩 + 微动效） |
| 弱化 | 进度条、题号 | 顶部窄条，信息型存在，不吸引注意力 |

#### 免费结果页

| 层级 | 内容 | 视觉权重 |
|------|------|---------|
| 第一眼 | MBTI 类型 + 灵魂伴侣标签 | 超大字号 + 标签色块，英雄区 |
| 第二眼 | 最佳匹配对象类型 | 次级视觉区，有吸引力但不抢主角 |
| 最重要动作 | 解锁完整灵魂伴侣报告按钮 | 强 CTA，视觉锚点 |
| 次要动作 | 分享按钮 | 存在但不与付费按钮竞争 |

#### 付费完整灵魂伴侣报告页

| 层级 | 内容 | 视觉权重 |
|------|------|---------|
| 第一眼 | 你的灵魂伴侣风格画像总结 | 首屏英雄区，大字 + 画像插图 |
| 第二眼 | 匹配 Top3 + 高风险 Top3 | 核心信息卡片区 |
| 最重要动作 | 生成 AI 深度解读按钮 | 页面底部强 CTA |
| 弱化 | 各模块间的过渡说明文字 | 极简，不堆砌 |

#### AI 深度解读页

| 层级 | 内容 | 视觉权重 |
|------|------|---------|
| 第一眼 | 灵魂伴侣核心驱动力 / 开篇金句 | 大字号引言式排版 |
| 第二眼 | 各解读段落 | 杂志式阅读节奏 |
| 最重要动作 | 分享按钮 | 阅读完成后的自然终点 |

### 14.3 页面构图定义

不同页面采用不同构图范式，严禁所有页面都用"卡片堆砌"：

| 页面 | 构图类型 | 说明 |
|------|---------|------|
| 首页 | 海报式 | 一张主视觉 + 一句话 + 一个按钮，像电影海报一样简洁有力 |
| 基础信息填写 | 编辑器式 | 纯净背景 + 居中表单，零干扰，像对话一样逐步填写 |
| MBTI 测试 | 编辑器式 | 大面积留白 + 居中题目 + 底部选项，沉浸专注 |
| 结果计算等待 | 海报式 | 全屏动效 + 滚动文案，仪式感 > 信息量 |
| 免费结果页 | 档案册式 | 个人档案卡 + 核心数据，像翻开一份关于自己的档案 |
| 完整灵魂伴侣报告 | 杂志式 | 长滚动，图文交替，有呼吸节奏，每个模块有独立视觉锚点 |
| AI 深度解读 | 杂志式 | 偏文字阅读体验，段落间留白充足，像读一封写给你的信 |
| 分享海报 | 海报式 | 单屏截图友好，核心结论 + 视觉符号 + 二维码 |

#### 构图禁忌

- 禁止满屏卡片——每屏最多 2 个独立视觉容器
- 禁止满屏边框——用留白和层级代替线框分隔
- 禁止满屏说明文字——每个模块最多一句解释
- 禁止无整体气场——每屏必须有一个视觉统治元素

### 14.4 极简内容策略

每一屏内容审查清单——只保留：

- 1 个主标题
- 1 句解释（最多 20 字）
- 1 个主按钮
- 1 个核心视觉锚点

执行规则：

- 如果删掉某段文字页面更好看，那段文字就不该存在
- 不写"欢迎来到""请您"等无意义客套
- 不在按钮旁重复解释按钮的功能
- 标签和短语优于完整句子
- 数据和结论优于描述和修饰

#### 配色体系

整体色彩策略：`10% 克制色彩`，大面积留白 + 极少量高饱和强调色。

| Token | 色值 | 用途 |
|-------|------|------|
| `color-bg-primary` | `#0A0A0F` | 主背景（深色模式主基调） |

| `color-bg-tertiary` | `#1A1A24` | 悬浮层、输入框底 |

| `color-danger` | `#FF453A` | 高风险冲突 |

色彩铁律：

- 强调色在一屏内只出现 1-2 处，绝不满屏铺
- 背景色用深色大面积铺底，营造星空 / 宇宙 / 神秘氛围
- 文字层级靠透明度区分，不靠换颜色
- 匹配度用语义色传达，不做彩虹式配色

|-------|-----|------|
| `shadow-sm` | `0 1px 2px rgba(0,0,0,0.3)` | 微浮起（标签） |

| `shadow-lg` | `0 8px 32px rgba(0,0,0,0.5)` | 弹窗、浮层 |
| `shadow-glow` | `0 0 24px rgba(124,92,252,0.25)` | 强调元素光晕（CTA、结果类型） |

#### 边框与分隔

- 不使用实色边框分隔内容
- 需要分隔时使用 `1px solid rgba(255,255,255,0.06)` 极低透明度线
- 优先使用留白和背景色差分隔模块
- 卡片不加边框，靠背景色层级区分

#### 动效节奏系统

| Token | 值 | 用途 |
|-------|-----|------|
| `duration-fast` | `150ms` | 按钮 hover、选项高亮 |

| `duration-slow` | `500ms` | 页面切换、结果揭晓 |
| `duration-dramatic` | `800-1200ms` | 仪式感场景（结果计算、AI 生成） |
| `easing-default` | `cubic-bezier(0.25, 0.1, 0.25, 1)` | 通用 |
| `easing-spring` | `cubic-bezier(0.34, 1.56, 0.64, 1)` | 弹性反馈 |

### 14.6 对比体系

高级感来自对比明确，不是把所有东西调成中间值：

| 对比维度 | 策略 |
|---------|------|
| 大与小 | 英雄标题 `36px` vs 辅助文字 `11px`，至少 3 倍字号差 |
| 疏与密 | 英雄区留白 `96px` vs 信息区间距 `16px`，呼吸感对比鲜明 |
| 亮与暗 | 强调色文字/按钮 vs 深色大面积背景，亮的地方真亮、暗的地方真暗 |
| 静与动 | 90% 的页面是静态排版，只在关键节点触发动效 |
| 粗与细 | 标题 `Bold 700` vs 正文 `Regular 400`，字重差必须明显 |
| 实与虚 | 核心数据实体色显示 vs 背景元素用半透明 / 模糊处理 |

### 14.7 动效设计规范

#### 动效只做三件事

1. **引导注意力**——告诉用户哪里先看
2. **反馈状态变化**——告诉用户操作生效了
3. **平滑过渡**——让页面切换不突兀

#### 各场景动效定义

| 场景 | 动效类型 | 说明 |
|------|---------|------|
| 首页进入 | 渐现 + 微上移 | 主视觉先出，标题随后，按钮最后，节奏间隔 `150ms` |
| 测试选项选中 | 缩放弹性 + 色彩过渡 | 选中项轻微放大 `scale(1.02)` + 强调色填充 |
| 题目切换 | 左右滑动 | 方向与进度一致，有物理位移感 |
| 进度条推进 | 宽度渐变 | 线性增长，不跳跃 |
| 结果计算页 | 序列帧文案 + 背景粒子 | 文案逐句渐现，背景有缓慢流动的星空粒子 |
| 结果揭晓 | 缩放弹入 + 光晕 | MBTI 类型从 `scale(0.8)` 弹入 `scale(1)` + `shadow-glow` |
| 滚动展开 | 交叉渐现 | 模块随滚动逐个浮现，`translateY(20px)` → `0` |
| AI 生成中 | 打字机效果 | 文字逐字/逐句出现，模拟"正在为你写"的感觉 |
| 支付成功 | 轻量庆祝 | 一次性微粒子扩散，克制的确认感 |

#### 动效禁忌

- 禁止全屏所有元素同时运动
- 禁止纯装饰性弹跳、漂浮、旋转
- 禁止超过 `1200ms` 的阻塞性动效
- 禁止动效阻断用户操作

### 14.8 核心视觉素材方向

#### 主视觉风格

- 星空 / 宇宙 / 星座意象作为背景基调
- 搭配抽象几何或 3D 元素表达人格类型
- 不使用写实人物照片，不使用低质量卡通
- 插画风格统一为：极简线条 + 渐变光效 + 半透明层叠

#### 分享海报规范

- 单屏比例 `3:4`（适配微信 / 小红书）
- 内容仅保留：MBTI 类型 + 灵魂伴侣标签 + 最佳匹配 + 产品二维码
- 字号放大至截图后清晰可读
- 背景使用深色 + 渐变光效，不使用白色底

### 14.9 字体方案

| 用途 | 字体 | 回退 |
|------|------|------|
| 中文正文 | `"PingFang SC"` | `"Noto Sans SC"`, `system-ui` |
| 英文 / MBTI 类型展示 | `"SF Pro Display"` | `"Inter"`, `sans-serif` |
| 数字 / 数据 | `"SF Mono"` | `"JetBrains Mono"`, `monospace` |

MBTI 四字母类型使用等宽英文字体，确保视觉对齐和辨识度。

### 14.10 响应式与适配

- 设计基准：`375px` 宽（iPhone SE / 标准移动端）
- 最大内容宽度：`428px`（大屏手机）
- 横屏不做特殊适配，锁定竖屏体验
- 安全区适配：底部 CTA 避开 iOS Home Indicator

## 15. 交互设计

### 15.1 交互总原则

- 低决策成本
- 高反馈密度
- 强节奏感
- 有仪式感但不拖沓

### 15.2 首页交互

- 首屏只保留 1 个主要 CTA
- 页面进入时有轻量文字和背景动效
- 不堆长解释和卖点列表

### 15.3 测试页交互

- 一屏一道题
- 顶部固定进度条
- 选项点击即高亮
- 支持上一题返回
- 自动保存当前进度

### 15.4 结果页交互

- 首屏直接出答案
- 结果揭晓控制在 `1 秒左右`
- 向下滚动逐步展开更深内容
- 关键位置都能触发分享与付费

### 15.5 支付后交互

- 支付成功立即反馈
- 立即展示完整灵魂伴侣报告
- AI 解读为后续 CTA，不阻塞阅读

### 15.6 AI 生成交互

- 用户点击后进入短暂过渡页
- 展示生成状态文案
- 成功后自动跳转 AI 报告
- 失败时展示重试按钮和说明

### 15.7 异常状态与错误处理设计

每个关键页面都必须定义异常态的用户体验，不允许出现白屏或无反馈状态。

#### 网络异常

| 场景 | 表现 | 处理 |
|------|------|------|
| 首页加载失败 | 纯色背景 + 居中重试按钮 + 一句话提示 | 点击重试刷新页面 |
| 答题提交网络断开 | Toast 提示"网络不稳定，答案已保存在本地" | 恢复后自动同步 |
| 结果计算超时 | 等待 `8s` 后展示"正在加倍努力计算" | `15s` 后展示重试按钮 |

#### 支付异常

| 场景 | 表现 | 处理 |
|------|------|------|
| 支付取消 | 回到报告页，不弹任何弹窗 | 付费按钮保持可点击 |
| 支付失败 | Toast 提示"支付未成功，请重试" | 保留订单 `15 分钟`，可重新支付 |
| 支付成功但回调延迟 | 展示"正在确认支付结果……" + 加载动画 | 前端轮询 `30s`，超时提示联系客服 |
| 重复支付 | 服务端幂等校验 | 已支付订单不重复扣款 |

#### AI 生成异常

| 场景 | 表现 | 处理 |
|------|------|------|
| AI 生成超时（`>15s`） | 文案切换为"分析内容比较丰富，再等一下" | `30s` 后展示重试按钮 |
| AI 生成失败 | 展示"生成遇到问题" + 重试按钮 | 最多重试 `2 次`，仍失败则提示稍后再来 |
| AI 返回内容异常 | 服务端校验输出结构 | 不合格内容不落库，触发重试 |

#### 通用兜底

- 所有页面 `5s` 无响应展示骨架屏，`15s` 无响应展示错误态
- 错误态视觉：与产品整体风格一致的插画 + 简短文案 + 操作按钮
- 禁止使用浏览器默认错误页或空白页

| 海报生成 | `< 5s` | Satori 渲染 + 上传 |

### 22.3 动效帧率

- 所有过渡动效保持 `60fps`
- 星空粒子背景不得导致低端机掉帧，低端机降级为静态渐变背景
- 测试页切题动效在低端机使用 `opacity` 替代 `transform`

## 23. 灰度发布与 A/B Test

### 23.1 灰度发布方案

- 基于 Cookie 中的 `userId` 做哈希分桶
- 支持按百分比控制灰度流量（如 10% / 50% / 100%）
- 灰度期间新旧版本共存，数据分开统计

### 23.2 A/B Test 能力

通过 PostHog Feature Flags 实现：

| 实验项 | 变量 | 说明 |
|-------|------|------|
| 定价测试 | `9.9` vs `12.9` vs `19.9` | 验证最优价格点 |
| 付费墙文案 | 方案 A vs 方案 B | 验证哪种文案转化率更高 |
| AI 模型 | `gpt-4o-mini` vs `gpt-4o` | 验证模型对满意度的影响 |
| 首页主视觉 | 插画 vs 3D | 验证哪种视觉风格起测率更高 |

### 23.3 实验流程

1. 在 PostHog 创建 Feature Flag
2. 前端 / 服务端读取 Flag 值决定展示方案
3. 埋点事件携带实验分组标识
4. 实验至少运行 `7 天` 或覆盖 `1000 UV` 后才做结论
5. 胜出方案全量上线，清理实验代码

## 25. 监控与告警

### 25.1 核心监控指标

| 指标 | 告警阈值 | 说明 |
|------|---------|------|
| AI 生成成功率 | `< 95%`（5 分钟窗口） | 可能是 OpenAI 服务异常 |
| AI 平均延迟 | `> 12s`（5 分钟窗口） | 接近超时阈值 |
| 支付回调成功率 | `< 98%`（1 小时窗口） | 可能存在支付通道问题 |
| 接口 5xx 错误率 | `> 1%`（5 分钟窗口） | 服务端异常 |
| 首页加载成功率 | `< 99%` | CDN 或部署问题 |
| 测试完成率日环比 | 下降 `> 20%` | 可能存在题目或流程问题 |

### 25.2 日志要求

| 模块 | 记录内容 |
|------|---------|
| AI 调用 | 请求 ID、模型、输入 token、输出 token、耗时、状态、错误信息 |
| 支付 | 订单 ID、金额、渠道、状态流转时间、回调原始数据 |
| 报告生成 | 用户 ID、MBTI 类型、匹配结果、计算耗时 |
| 异常 | 错误栈、请求上下文、用户标识 |

### 25.3 告警通道

- V1 使用飞书 / 钉钉 Webhook 推送告警
- 告警分级：P0（立即处理） / P1（1 小时内） / P2（当日内）

| 级别 | 场景 |
|------|------|
| P0 | 支付通道不可用、全站 5xx、AI 服务完全不可用 |
| P1 | AI 成功率低于阈值、支付回调延迟、单接口响应超时 |
| P2 | 埋点数据缺失、非核心页面加载慢、日活异常波动 |
